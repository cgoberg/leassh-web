Homelab Monitoring with Leassh: One Tool for Your Servers and Your Family

Carl-Gustav Öberg July 23, 2026 6 min read

I built Leassh because I had a homelab problem that I couldn't solve with the right tool. Not because the tools were bad, but because every tool I tried solved only half my problem.

I had a handful of machines at home — a Proxmox node, a NAS, a couple of gaming PCs my kids use — and I wanted to know what was happening on them without spending a weekend building dashboards. I also wanted to know who was logged in and what they were doing, because they're my kids and it's my job as a parent to care.

The monitoring tools out there handled the infrastructure side. The parental control tools handled the family side. None of them did both, and most of them shipped data to the cloud — which felt wrong for something running in my own home.

So I built Leassh around three principles: SSH-first, self-hosted, and dual-purpose.

SSH-First Monitoring

The core of Leassh is simple: it SSHs into your machines, runs commands, and parses the output. That's it. No agents to install on the monitored machines, no extra ports to open, no sidecar containers. Every Linux, macOS, and Windows machine on your network that already answers SSH can be monitored.

This is the opposite of the traditional monitoring model where every machine runs an agent that phones home to a central server. The agent model works, but it introduces failure modes I didn't want in my homelab: agent crashes, stale data after a reboot, firewall rules for the agent port, and the cognitive overhead of keeping agents updated across a fleet of machines running different OS versions.

With SSH-first, the intelligence lives on the monitoring side. The monitored machines don't know they're being monitored — they're just answering commands over a connection that already exists.

Why SSH for Homelab Monitoring

SSH is already available on every Linux and macOS machine by default, and on Windows since 2019. It's authenticated, encrypted, and you already manage the keys. Using SSH for monitoring means you get security and reachability without adding a single new service.

Leassh maintains persistent SSH sessions to each node in your fleet. Every minute or so, it runs a set of OS-appropriate commands to collect CPU load, memory pressure, disk usage, GPU utilization, active users, idle time, and running processes. The results are correlated across probes, so the system can detect trends like disk usage growing faster than expected or a process suddenly consuming more resources than usual.

Self-Hosted by Design

The monitoring data stays in your home. Leassh doesn't ship screenshots, process lists, or user activity to any cloud service. The only external communication is the license validation — a single check against our Supabase instance to verify your key is active.

This is a board-level decision, not a feature we might add later. The core promise of Leassh is that the people running it own their data. For homelab users, this means no unexpected telemetry. For parents, it means their child's screenshots and screen time logs live on their own hardware, not on a third-party server.

The architecture reflects this:

  • Leassh server runs on your own machine (the .127 in my case) via standalone.js, a Node.js HTTP server that serves the dashboard and bridges to the Rust binary through JSON-RPC IPC
  • Dashboard is a local web interface served on port 8383 — accessible from your LAN, not exposed to the internet unless you configure port forwarding
  • Database for license keys is Supabase (external), but monitoring data is held in memory and optional local logs — nothing in Supabase contains your fleet metrics

If you've been following the homelab space, you know the trend is toward self-hosted alternatives for everything — Pi-hole instead of cloud DNS, Gitea instead of GitHub, Nextcloud instead of Google Drive. Leassh fits into this philosophy: a monitoring tool that respects the boundary between your network and the rest of the internet.

Dual-Purpose: Infrastructure and Family

The part of Leassh that surprises people is the family side. It's not a bolt-on or a marketing angle — it grew out of the same problem that started the project.

I was already SSH-ing into my kids' gaming PCs to check what they were doing — what process was hogging the GPU, whether they were actually "doing homework" or just staring at Steam. Leassh automates that SSH session and adds structure to it.

Screen Time Tracking

Leassh tracks active window time per user on each machine. The system knows when a user is idle (no mouse or keyboard input for the configured threshold), when they're actively using an application, and which processes are consuming resources. This feeds into the screen time engine, which enforces rules you configure.

The rules are simple: you define allowed hours, maximum daily screen time, and application-specific limits. When a limit is reached, Leassh can send a notification, lock the session, or even shut down the machine — all configurable per user and per machine.

Screenshots

Leassh captures periodic screenshots over SSH, using platform-appropriate tools: scrot or grim on Linux, screencapture on macOS, and a PowerShell-based approach on Windows (which handles the Session 0 desktop issue automatically). Screenshots are stored locally and viewable in the dashboard.

For homelab users who care about security, the screenshot data never leaves the local machine. For parents, it provides a visual record of what their kids are doing on the computer.

Process Monitoring

Every probe cycle, Leassh collects the running process list from each machine. The system builds a baseline of "normal" processes and flags new or unusual ones. This is useful for homelab ops — catching a runaway process or an unexpected service — but it's also useful for parents who want to know if a supposedly "educational" window is actually Discord in a browser tab.

The Dashboard Experience

The Leassh dashboard is a local web interface served by standalone.js. It shows all nodes at a glance with status indicators, CPU/RAM/GPU usage bars, active users, idle timers, and last-seen timestamps. The fleet view is a single page; the family view drills into per-user activity on specific machines.

The dashboard auto-refreshes, so you can leave it open on a second monitor or a tablet and see your homelab status update in real time. It's not a Grafana-level visualization — it's simpler by design. The goal is to give you the information you need in under ten seconds, not to build custom panels for every metric.

nas ONLINE CPU 3% disk 812GB free
gaming-3 ONLINE user: felix, GPU 91% (Fortnite)
gaming-1 IDLE (no users, 47min)

Rules and Automation

Beyond visibility, Leassh lets you automate actions based on conditions you define. The rules engine supports:

  • "Kill process X on machine Y when GPU usage exceeds Z%" — useful for reclaiming a gaming PC's GPU for inference work
  • "Lock user session after N hours of screen time" — the core parental control feature
  • "Notify me when disk usage crosses 85%" — basic infrastructure alerting
  • "Send a shutdown signal to all gaming PCs at 10 PM on weekdays" — bedtime enforcement

Rules are defined in the same fleet.yaml that lists your machines. There's no separate alerting configuration, no webhook URLs to wire up. The rules are evaluated locally by the Leassh server, and the actions are executed over the SSH connection.

Platform Support

Leassh monitors machines across three major desktop platforms:

Platform Metric Coverage Screenshots Notes
Linux (x86_64) Full Yes Native SSH, scrot/grim for screenshots
macOS (Apple Silicon) Full Yes Native SSH, screencapture
Windows (x86_64) Partial Yes* OpenSSH Server, PowerShell screenshots*

* Windows screenshots require a scheduled task in the interactive user session. Leassh creates this automatically on first connect.

The command registry — a JSON file that maps metric names to OS-specific shell commands — is extensible. If you have a machine running a niche Linux distribution or a custom build, you can add command variants to the registry without recompiling the Leassh binary.

What Leassh Is Not

Leassh is not a replacement for Prometheus or Grafana if you have 50+ nodes and need sub-second metric resolution. It's designed for the homelab sweet spot: 3–15 machines where you want a quick overview without the overhead of a full monitoring stack.

It's not a SaaS product. There's no cloud-hosted dashboard, no multi-tenant data plane. You run the Leassh server yourself, on hardware you own.

And it's not a general-purpose remote desktop tool. The screenshots are periodic snapshots for monitoring purposes, not a way to take control of your kids' machine in real time.

Getting Started

Setting up Leassh for homelab monitoring takes about ten minutes:

  1. Install the binary — download the Leassh server for your platform from our releases page and place it on a machine that has SSH access to your fleet
  2. Write fleet.yaml — list your machines with their SSH details, OS type, and probe intervals
  3. Run the server — start standalone.js, which serves the dashboard and bridges to the Rust binary via IPC
  4. Open the dashboard — point your browser to http://localhost:8383/fleet and start seeing your machines come online
